# ESP32系列芯片，ESP-IDF，MacOS开发常规Skill

适用场景：
- Mac 平台
- 官方 `ESP-IDF`
- 新的 `ESP32` 板子，从零开始
- 目标是尽快跑通“编译 -> 烧录 -> 串口 -> 设备自检 -> 迭代”

## 核心原则

1. 先做最小闭环，不要一开始就写完整功能。
2. 每次烧录后，先让设备自己报状态，尽量少靠人工目测。
3. 把串口日志和操作记录都留在项目目录里，方便回看。
4. 任何硬件不确定项，先做“可观测自检”，不要先假设。

## 推荐流程

### 0. 环境

- 使用官方 `ESP-IDF` Mac 安装方式。
- 激活环境脚本后再构建：
  - `source "/Users/ky/.espressif/tools/activate_idf_v6.0.1.sh"`
- 项目内保留脚本：
  - `scripts/build.sh`
  - `scripts/flash.sh`
  - `scripts/monitor.sh`
  - `scripts/capture-serial.sh`

### 1. 先跑通闭环

- 最小程序只做两件事：
  - 串口打印 `hello`
  - 确认自动下载/复位能烧录
- 串口日志必须保存到 `docs/serial-logs/`
- 第一次闭环成功后，再做硬件自检，不要直接进大功能


### 2. 减少人工测试的经验

- 每个硬件点都要有“串口可验证结果”，不要只靠眼睛看。
- 固件启动后主动打印：
  - 芯片型号
  - Flash 大小
  - PSRAM 是否识别
  - 关键 GPIO 配置
  - ROM 是否加载成功
- 每个阶段都做“自动报告”：
  - 设备启动成功
  - 资源分配成功
  - ROM 加载成功
  - 主循环已进入
- 把测试分两层：
  - 机器可自动判断的放串口
  - 只能人工感知的只保留最后确认

### 3. 烧录后尽量少人工测试

- 烧录后先看串口，不要先盯屏幕。
- 在固件里加入明确的阶段日志：
  - `boot`
  - `tft init ok`
  - `buttons ok`
  - `audio ok`
  - `rom loaded`
  - `main loop running`
- 出现问题时，优先用日志缩小范围：
  - 初始化失败
  - 资源分配失败
  - 中断/看门狗
  - 显示方向错误
  - 音频节奏错误
- 设备能自己说清楚状态，就少做重复插拔和盲测

### 4. 分区和 ROM

- 如果不做 SD 卡，优先把 `factory` app 分区开大。
- 先算剩余空间，再决定 ROM 是：
  - 直接 embed 到 app
  - 还是单独做 data 分区
- 先固定一个可跑的 ROM，再扩展多 ROM 菜单

### 5. 文件组织

- `Hardware/`：手册、照片、引脚总结
- `assets/`：当前使用的 ROM 或静态资源
- `docs/serial-logs/`：串口日志
- `docs/codex-operation-records/`：每次关键改动记录
- `Skills/`：本次开发方法总结，留给下次 ESP32系列芯片 项目复用

## 这次项目里可复用的结论

- 官方 `ESP-IDF` 在 Mac 上从零搭建是可行的，关键是把激活脚本、构建、烧录、监视封装成项目脚本。
- 没有 BOOT 按钮也没关系，只要串口下载链路正常，自动复位仍可用。
- 先用最小程序确认闭环，再做硬件自检，会明显减少后续返工。
- 屏幕、按键、ADC、蜂鸣器都适合做成“串口报告式”自检。
- GBC 模拟器这种大功能，先用成熟开源核心移植，不要先造轮子。
- 音频如果硬件受限，先用可控的固定反馈音，后面再追求更像原机的效果。

## 下次新板子时的起步清单

1. 装官方 `ESP-IDF`
2. 跑 `hello`
3. 保存串口日志
4. 扫手册确认引脚
5. 写硬件自检
6. 再做目标功能移植
